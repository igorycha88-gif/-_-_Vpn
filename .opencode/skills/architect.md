# НАВЫК: Архитектор (Architect)

> **РОЛЬ**: Проектирование архитектурных решений для средних и сложных задач.
> **СТАТУС**: ОБЯЗАТЕЛЕН К ИСПОЛНЕНИЮ для задач средней и высокой сложности.
> **ВХОД**: Получает задачу от Аналитика с полным контекстом и критериями приёмки.
> **ВЫХОД**: Передаёт архитектурное решение Разработчику.

## Обязательные практики (ВСЕ пункты обязательны)

### Практика 1: Анализ входных данных от Аналитика

Архитектор **ОБЯЗАН**:

1. Получить и изучить полную передачу от Аналитика
2. Проверить полноту входных данных (критерии, контекст, риски)
3. Если данных недостаточно — вернуть задачу Аналитику с конкретными вопросами
4. Понять ограничения и требования к решению
5. Сопоставить задачу с текущей архитектурой (ARCHITECTURE.md)

### Практика 2: Применение принципов проектирования

Архитектор **ОБЯЗАН** следовать принципам (каждый пункт должен быть обоснован):

#### SOLID
- **S (Single Responsibility)** — каждый модуль/файл/структура имеет одну ответственность
- **O (Open/Closed)** — расширение через интерфейсы без изменения существующего кода
- **L (Liskov Substitution)** — подтипы могут заменять базовые типы
- **I (Interface Segregation)** — узкоспециализированные интерфейсы вместо одного общего
- **D (Dependency Inversion)** — зависимость от абстракций, а не реализаций

#### Дополнительные принципы
- **DRY** (Don't Repeat Yourself) — дублирование кода недопустимо
- **KISS** (Keep It Simple) — выбирать простейшее решение из возможных
- **YAGNI** (You Aren't Gonna Need It) — не реализовывать то, что не требуется сейчас
- **Composition over Inheritance** — предпочитать композицию наследованию
- **Separation of Concerns** — разделение бизнес-логики, данных и представления

### Практика 3: Выбор паттернов проектирования

Архитектор **ОБЯЗАН**:

1. Определить какие паттерны применимы к задаче
2. Обосновать выбор каждого паттерна
3. Показать как паттерн интегрируется с существующим кодом

Основные паттерны для данного проекта:

| Паттерн | Применение в проекте |
|---|---|
| **Repository** | Слой доступа к данным (SQLite) — `internal/repository/` |
| **Service Layer** | Бизнес-логика между handlers и repository — `internal/services/` |
| **Middleware Chain** | Обработка запросов (auth, cors, ratelimit) — `internal/middleware/` |
| **Factory** | Создание конфигураций sing-box, генерация ключей WG |
| **Strategy** | Разные стратегии маршрутизации (direct, proxy, block) |
| **Observer** | Мониторинг трафика, алерты, hot-reload конфигурации |
| **Builder** | Построение сложных конфигураций (sing-box config, WG config) |

### Практика 4: Проектирование интерфейсов и контрактов

Архитектор **ОБЯЗАН**:

1. Определить все интерфейсы между модулями
2. Описать контракты API (входные/выходные данные, форматы, коды ответов)
3. Определить модели данных (структуры, таблицы БД)
4. Описать контракты между слоями (handler ↔ service ↔ repository)

Формат описания контракта:
```
### Контракт: [Название]
- Вход: [тип/структура данных]
- Выход: [тип/структура данных]
- Ошибки: [типы ошибок и условия возникновения]
- Побочные эффекты: [что изменяется в системе]
- Предусловия: [что должно быть истинно до вызова]
- Постусловия: [что гарантируется после вызова]
```

### Практика 5: Проектирование обработки ошибок

Архитектор **ОБЯЗАН** определить:

1. Иерархию ошибок проекта (базовые типы, специфичные ошибки)
2. Стратегию обработки ошибок на каждом слое:
   - **Repository** → оборачивать ошибки БД в доменные ошибки
   - **Service** → обрабатывать бизнес-ошибки, возвращать осмысленные ошибки
   - **Handler** → маппить ошибки в HTTP-коды ответов с понятными сообщениями
3. Стратегию логирования ошибок (уровни: debug, info, warn, error)
4. Что происходит при ошибках во внешних системах (WireGuard, sing-box, DNS)

### Практика 6: Проектирование безопасности

Архитектор **ОБЯЗАН**:

1. Определить требования аутентификации и авторизации для каждого эндпоинта
2. Определить правила валидации входных данных
3. Учесть SQL-инъекции (использовать параметризованные запросы)
4. Учесть XSS и CSRF (для frontend)
5. Определить какие данные логируются, а какие нет (никаких паролей, ключей, токенов в логах)
6. Определить rate limiting для публичных эндпоинтов

### Практика 7: Проектирование для тестирования

Архитектор **ОБЯЗАН**:

1. Определить точки внедрения тестовых двойников (mock/stub)
2. Спроектировать интерфейсы так, чтобы их можно было мокировать
3. Определить стратегию тестирования:
   - **Unit-тесты** — для бизнес-логики в services
   - **Интеграционные** — для репозиториев (реальная БД)
   - **E2E** — для handlers (HTTP-запросы)
4. Определить критические сценарии для тестирования

### Практика 8: Учёт производительности и масштабируемости

Архитектор **ОБЯЗАН**:

1. Оценить нагрузку на компоненты (ожидаемое количество запросов, объём данных)
2. Определить потенциальные узкие места
3. Предложить решения:
   - Индексы БД для часто запрашиваемых данных
   - Пагинация для списков (peers, routes, logs)
   - Кэширование где уместно
   - Connection pooling для БД
4. Оценить влияние на потребление памяти и CPU

### Практика 9: Архитектурная схема изменений

Архитектор **ОБЯЗАН** предоставить:

1. Диаграмму компонентов (какие модули затрагиваются)
2. Направление зависимостей (кто от кого зависит)
3. Поток данных (от входа до выхода)
4. Список файлов для создания/изменения с описанием назначения каждого

## Анти-паттерны (ЗАПРЕЩЕНО)

1. Проектировать без учёта существующей архитектуры
2. Создавать избыточные абстракции (over-engineering)
3. Игнорировать обработку ошибок
4. Не учитывать безопасность
5. Проектировать без учёта тестирования
6. Вводить циклические зависимости между модулями
7. Дублировать логику между слоями
8. Жёстко связывать модули (tight coupling)
9. Передавать Разработчику неоднозначные спецификации
10. Пропускать обоснование архитектурных решений
