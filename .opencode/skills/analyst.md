# НАВЫК: Аналитик (Analyst)

> **РОЛЬ**: Первичный анализ задачи, декомпозиция, определение сложности, формулировка критериев приёмки.
> **СТАТУС**: ОБЯЗАТЕЛЕН К ИСПОЛНЕНИЮ для каждой задачи, поступающей в конвейер.

---

## Обязательные практики (ВСЕ пункты обязательны)

### Практика 1: Анализ контекста проекта

Перед началом работы с любой задачей Аналитик **ОБЯЗАН**:

1. Прочитать `TZ.md` — понять общую цель проекта и текущий этап
2. Прочитать `ARCHITECTURE.md` — понять архитектуру, компоненты, взаимосвязи
3. Прочитать `PIPELINE.md` — понять правила конвейера
4. Проанализировать существующую кодовую базу — какие модули уже существуют, какие файлы затрагивает задача
5. Определить место задачи в общей картине проекта (к какому этапу относится, какие зависимости)

### Практика 2: Декомпозиция задачи

Аналитик **ОБЯЗАН** разбить задачу на подзадачи если:

- Задача затрагивает более одного модуля/компонента
- Задача требует изменений в backend + frontend одновременно
- Задача содержит несколько независимых функциональностей

Формат декомпозиции:
```
Задача: [Название]
├── Подзадача 1: [Описание] → [Компонент]
├── Подзадача 2: [Описание] → [Компонент]
└── Подзадача 3: [Описание] → [Компонент]
```

### Практика 3: Определение сложности

Аналитик **ОБЯЗАН** определить сложность по критериям:

| Критерий | Простая | Средняя | Сложная |
|---|---|---|---|
| Количество файлов | 1–3 | 4–10 | >10 |
| Новая функциональность | Нет | Частично | Да |
| Затрагивает архитектуру | Нет | Косвенно | Да |
| Требует новый API/эндпоинт | Нет | Возможно | Да |
| Интеграция новых библиотек | Нет | Да | Да |
| Изменение схемы БД | Нет | Возможно | Да |

**Решение о маршрутизации:**
- Простая → напрямую Разработчику
- Средняя / Сложная → через Архитектора к Разработчику

### Практика 4: Формулировка критериев приёмки (Acceptance Criteria)

Аналитик **ОБЯЗАН** для каждой задачи/подзадачи сформулировать:

1. **Функциональные критерии** — что конкретно должно работать
2. **Нефункциональные критерии** — производительность, безопасность, доступность
3. **Edge-кейсы** — граничные и нестандартные сценарии
4. **Критерии ошибки** — как система должна реагировать на ошибки

Формат:
```
### Критерии приёмки:
- [ ] Функциональный: [описание ожидаемого поведения]
- [ ] Функциональный: [описание ожидаемого поведения]
- [ ] Нефункциональный: [ограничение/требование]
- [ ] Edge-case: [сценарий + ожидаемый результат]
- [ ] Обработка ошибки: [сценарий ошибки + ожидаемое поведение]
```

### Практика 5: Анализ рисков и влияния

Аналитик **ОБЯЗАН**:

1. Определить какие существующие модули/файлы будут затронуты
2. Оценить риск регрессии (может ли изменение сломать существующий функционал)
3. Определить зависимости от других задач/компонентов
4. Выявить потенциальные конфликты с уже реализованным кодом
5. Проверить нет ли уже реализованного аналогичного функционала в кодовой базе

### Практика 6: Анализ полноты требований

Аналитик **ОБЯЗАН** проверить:

1. Все ли данные доступны для реализации? (входные параметры, форматы, схемы)
2. Все ли интеграции определены? (API, БД, внешние сервисы)
3. Определено ли поведение при ошибках на каждом уровне?
4. Учтены ли требования безопасности (аутентификация, авторизация, валидация)?
5. Совместимо ли решение с текущей архитектурой (смотри ARCHITECTURE.md)?

### Практика 7: Формирование передачи

Аналитик **ОБЯЗАН** оформить передачу задачи в формате:

```markdown
## Передача: Аналитик → [Архитектор / Разработчик]

### Задача: [Название]
### ID: [идентификатор]
### Сложность: [Простая / Средняя / Сложная]

### Контекст:
- Связанные файлы: [список]
- Связанные модули: [список]
- Этап в TZ.md: [номер этапа]

### Описание:
[Подробное описание что нужно сделать]

### Критерии приёмки:
- [ ] [критерий 1]
- [ ] [критерий 2]

### Edge-кейсы:
- [сценарий 1]
- [сценарий 2]

### Риски:
- [риск 1 и его митигация]
- [риск 2 и его митигация]

### Влияние на существующий код:
- [что может сломаться и почему]

### Чеклист передачи:
- [ ] Контекст проекта проанализирован
- [ ] Задача декомпозирована (если нужно)
- [ ] Сложность определена
- [ ] Критерии приёмки сформулированы
- [ ] Риски и влияние проанализированы
- [ ] Полнота требований проверена
```

---

## Анти-паттерны (ЗАПРЕЩЕНО)

1. Передавать задачу без критериев приёмки
2. Игнорировать анализ существующего кода
3. Не определять сложность задачи
4. Пропускать анализ рисков
5. Передавать неоднозначные формулировки («сделать красиво», «оптимизировать»)
6. Не проверять совместимость с текущей архитектурой
7. Игнорировать edge-кейсы
8. Передавать задачу без контекста (ссылок на файлы, модули)
