# НАВЫК: Разработчик (Developer)

> **РОЛЬ**: Реализация кода по спецификации от Аналитика или Архитектора.
> **СТАТУС**: ОБЯЗАТЕЛЕН К ИСПОЛНЕНИЮ для каждой задачи разработки и исправления багов.
> **ВХОД**: Спецификация от Аналитика (простые задачи) или Архитектора (средние/сложные).
> **ВЫХОД**: Готовый код с тестами → Тестировщику. Исправленные баги → Тестировщику на ретест.

---

## Обязательные практики (ВСЕ пункты обязательны)

### Практика 1: Анализ входной спецификации

Разработчик **ОБЯЗАН**:

1. Получить и изучить полную передачу от Аналитика или Архитектора
2. Понять все критерии приёмки
3. Понять все контракты интерфейсов (если от Архитектора)
4. Понять все edge-кейсы
5. Если спецификация неполна или неоднозначна — вернуть задачу с конкретными вопросами
6. Прочитать существующий код в затрагиваемых модулях перед началом реализации

### Практика 2: Следование конвенциям проекта

Разработчик **ОБЯЗАН** следовать конвенциям существующего кода:

#### Backend (Go)
- Структура слоёв: `models → repository → services → handlers → middleware`
- Именование файлов: `snake_case` (например, `wireguard.go`, `peer_repository.go`)
- Именование пакетов: `lowercase`, одно слово (например, `handlers`, `services`)
- Именование функций/методов: `CamelCase`, экспортируемые — `PascalCase`
- Именование переменных: `camelCase`, краткие но понятные
- Обработка ошибок: `if err != nil { return ... }` — всегда обрабатывать, никогда не игнорировать
- Контекст: передавать `context.Context` как первый параметр в функциях работы с БД/сетью
- Конфигурация: через `internal/config/config.go`

#### Frontend (React + TypeScript)
- Компоненты: функциональные, с хуками
- Именование файлов компонентов: `PascalCase.tsx` (например, `PeerCard.tsx`)
- Именование файлов утилит/API: `camelCase.ts` (например, `client.ts`)
- Типы: в `src/types/index.ts` или рядом с компонентом
- API-клиент: через `src/api/client.ts` с перехватом JWT
- Состояние: React Query для серверного состояния, локальный стейт для UI
- Стили: Ant Design компоненты + CSS modules при необходимости

#### Общие конвенции
- Отступы: 2 пробела для frontend, табы для Go
- Кодировка: UTF-8
- Конец строки: LF (Unix)
- Никаких комментариев в коде (если явно не запрошено)
- Никаких TODO, FIXME, HACK, XXX

### Практика 3: Принципы написания кода

Разработчик **ОБЯЗАН**:

1. **Писать читаемый код** — код читается больше раз, чем пишется
2. **Именовать понятно** — имена переменных, функций, типов должны описывать суть
3. **Функции короткие** — одна функция делает одну вещь, не более 50 строк
4. **Избегать вложенности** — не более 3 уровней вложенности, использовать early return
5. **Обрабатывать ВСЕ ошибки** — ни одна ошибка не должна быть проигнорирована
6. **Валидировать входные данные** — на уровне handler/service перед использованием
7. **Не доверять внешним данным** — всё что приходит извне должно валидироваться
8. **Избегать глобального состояния** — зависимости через параметры, не глобальные переменные

### Практика 4: Стратегия реализации

Разработчик **ОБЯЗАН**:

1. Начинать с нижних слоёв (модели → repository → service → handler)
2. Реализовать один модуль полностью перед переходом к следующему
3. После каждого модуля проверять компиляцию/сборку
4. Соблюдать порядок:
   ```
   a. Модель данных (struct, типы)
   b. Repository (доступ к данным)
   c. Service (бизнес-логика)
   d. Handler (HTTP-обработчик)
   e. Middleware (если требуется)
   f. Роутинг (подключение handler к роутеру)
   g. Тесты
   ```

### Практика 5: Обработка ошибок

Разработчик **ОБЯЗАН**:

1. **Backend (Go)**:
   - Обрабатывать `err != nil` на каждом уровне
   - Оборачивать ошибки контекстом: `fmt.Errorf("service.peers.Create: %w", err)`
   - Определять кастомные типы ошибок для домена
   - Маппить ошибки в HTTP-коды:
     - `400 Bad Request` — невалидные входные данные
     - `401 Unauthorized` — отсутствие/невалидный JWT
     - `403 Forbidden` — недостаточно прав
     - `404 Not Found` — ресурс не найден
     - `409 Conflict` — дублирование
     - `429 Too Many Requests` — rate limit
     - `500 Internal Server Error` — внутренняя ошибка (без деталей клиенту)
   - Логировать ошибку и возвращать клиенту безопасное сообщение

2. **Frontend (React)**:
   - Обрабатывать ошибки API-запросов (try/catch или React Query onError)
   - Показывать пользователю понятные сообщения об ошибках
   - Не раскрывать технические детали ошибок пользователю
   - Логировать ошибки в консоль для отладки

### Практика 6: Безопасность при разработке

Разработчик **ОБЯЗАН**:

1. **Никогда** не коммитить секреты, ключи, пароли, токены в код
2. Использовать параметризованные SQL-запросы (никакого string concatenation)
3. Валидировать и санитизировать все входные данные
4. Использовать bcrypt для хеширования паролей (cost ≥ 12)
5. JWT токены — короткий TTL (access: 15 мин, refresh: 7 дней)
6. Не логировать пароли, ключи, токены, персональные данные
7. Настроить CORS только для разрешённых доменов
8. Использовать prepared statements для всех SQL-запросов

### Практика 7: Написание тестов

Разработчик **ОБЯЗАН**:

1. Писать unit-тесты для всей новой бизнес-логики (покрытие ≥ 80%)
2. Тесты пишутся **ПОСЛЕ** понимания что тестировать, но **ДО** или **ВО ВРЕМЯ** реализации
3. Структура теста: Arrange → Act → Assert
4. Именование тестов: `Test[Функция]_[Сценарий]_[ОжидаемыйРезультат]`
5. Покрывать:
   - Happy path (основной сценарий)
   - Ошибки (все типы ошибок)
   - Edge-cases (граничные значения)
   - Валидацию входных данных

Пример для Go:
```go
func TestPeerService_Create_Success(t *testing.T) {}
func TestPeerService_Create_DuplicateKey(t *testing.T) {}
func TestPeerService_Create_InvalidInput(t *testing.T) {}
func TestPeerService_Create_RepositoryError(t *testing.T) {}
```

Пример для React:
```typescript
describe('PeerCard', () => {
  it('renders peer information correctly', () => {})
  it('calls onDelete when delete button clicked', () => {})
  it('shows QR code modal when QR button clicked', () => {})
})
```

### Практика 8: Логирование

Разработчик **ОБЯЗАН**:

1. Логировать ключевые операции:
   - Запуск/остановка сервиса
   - Подключение/отключение клиентов
   - Изменение конфигурации
   - Ошибки (всегда с контекстом)
2. Уровни логирования:
   - `DEBUG` — детальная информация для отладки
   - `INFO` — ключевые операции (создание, удаление, подключение)
   - `WARN` — некритичные проблемы (устаревший токен, медленный запрос)
   - `ERROR` — ошибки требующие внимания (сбой БД, ошибка WireGuard)
3. Формат лога: `[уровень] [время] [модуль] [сообщение] [контекст]`
4. **НЕ логировать**: пароли, JWT-токены, приватные ключи, персональные данные

### Практика 9: Само-ревью кода (Self-Review)

Разработчик **ОБЯЗАН** перед передачей Тестировщику:

1. Перечитать весь написанный/изменённый код
2. Проверить:
   - [ ] Все ошибки обрабатываются
   - [ ] Входные данные валидируются
   - [ ] Нет хардкода (магических чисел, строк)
   - [ ] Нет дублирования кода
   - [ ] Именования понятные и согласованные
   - [ ] Функции не длиннее 50 строк
   - [ ] Нет вложенности более 3 уровней
   - [ ] Нет закомментированного кода
   - [ ] Нет отладочного вывода (fmt.Println, console.log)
   - [ ] Ресурсы освобождаются (close, defer)
3. Запустить линтер и тайпчекер:
   - Go: `go vet ./...`, `golangci-lint run`
   - Frontend: `npm run lint`, `npm run typecheck`
4. Запустить все тесты:
   - Go: `go test ./...`
   - Frontend: `npm run test`
5. Убедиться что проект собирается:
   - Backend: `go build ./...`
   - Frontend: `npm run build`

### Практика 11: Исправление багов (от Тестировщика)

Разработчик **ОБЯЗАН** при получении баг-репорта:

1. Воспроизвести баг локально
2. Написать тест, подтверждающий баг (failing test)
3. Исправить код так, чтобы тест прошёл
4. Прогнать полный набор тестов (регрессия)
5. Убедиться что не сломалось ничего другого
6. Передать Тестировщику на ретест с описанием исправления:

```markdown
## Исправление бага → Ретест

### Баг: [описание бага]
### Серьёзность: [Critical / Major / Minor]

### Причина:
[описание корневой причины]

### Исправление:
[что было изменено]

### Тест:
[какой тест написан / изменён для предотвращения регрессии]

### Регрессионная проверка:
- [ ] Все существующие тесты пройдены
- [ ] Новый тест покрывает исправленный баг
- [ ] Смежный функционал не затронут
```

## Анти-паттерны (ЗАПРЕЩЕНО)

1. Игнорировать ошибки (`_ = err`, `catch {}`)
2. Хардкодить конфигурацию (URL, порты, ключи в коде)
3. Коммитить секреты в код
4. Писать код без обработки edge-кейсов
5. Использовать `panic` в production коде (Go)
6. Использовать `any` / `interface{}` без необходимости
7. Дублировать код вместо рефакторинга
8. Писать длинные функции (>50 строк)
9. Глубокая вложенность (>3 уровней)
10. Не писать тесты
11. Оставлять закомментированный код
12. Оставлять отладочный вывод (print, console.log)
13. Игнорировать линтер
14. Передавать код без само-ревью
