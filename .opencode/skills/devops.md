# НАВЫК: DevOps

> **РОЛЬ**: Финальная сборка, интеграционная проверка, подведение итогов итерации.
> **СТАТУС**: ОБЯЗАТЕЛЕН К ИСПОЛНЕНИЮ после успешного тестирования.
> **ВХОД**: Подтверждённый отчёт о тестировании от Тестировщика (багов нет).
> **ВЫХОД**: Пересобранные сервисы + итоговый отчёт итерации.

---

## Обязательные практики (ВСЕ пункты обязательны)

### Практика 1: Анализ входных данных

DevOps **ОБЯЗАН**:

1. Получить отчёт о тестировании от Тестировщика с подтверждением отсутствия багов
2. Получить список всех изменённых файлов от Разработчика
3. Понять какие сервисы затронуты (backend, frontend, landing, nginx, sing-box)
4. Изучить `docker-compose.yml` и структуру проекта

### Практика 2: Полная пересборка сервисов

DevOps **ОБЯЗАН** пересобрать **ВСЕ** затронутые сервисы локально:

```bash
# 1. Остановить все запущенные контейнеры
docker compose down

# 2. Очистить старые образы (для чистоты сборки)
docker compose build --no-cache

# 3. Запустить все сервисы
docker compose up -d

# 4. Проверить статус контейнеров
docker compose ps
```

Для каждого сервиса проверить:
- [ ] Контейнер запустился (статус `Up`)
- [ ] Нет ошибок в логах контейнера
- [ ] Порт(ы) пробрасываются корректно
- [ ] Healthcheck проходит (если настроен)

### Практика 3: Интеграционная проверка

DevOps **ОБЯЗАН** выполнить интеграционную проверку всех сервисов в сборе:

#### 3.1. Проверка связки сервисов

```
Nginx (:443)
  ├── / → Landing Page (статический сайт)
  ├── /api/v1/* → Go Backend API (:8080)
  └── /admin/* → React SPA (:3000)
```

Проверки:
- [ ] Landing Page доступен по корневому URL
- [ ] API Backend отвечает на `/api/v1/` эндпоинты
- [ ] Frontend SPA загружается по `/admin/`
- [ ] CORS настроен корректно между Frontend и Backend
- [ ] Статические ресурсы Frontend загружаются (JS, CSS)

#### 3.2. Проверка Docker Compose

- [ ] `docker compose up -d` — все сервисы стартуют без ошибок
- [ ] `docker compose ps` — все сервисы в статусе Up
- [ ] `docker compose logs [service]` — нет ошибок в логах
- [ ] Контейнеры могут общаться по сети
- [ ] Volumes монтируются корректно
- [ ] Переменные окружения подхватываются

#### 3.3. Проверка нового функционала

- [ ] Конкретный функционал из задачи работает в полной сборке
- [ ] API-эндпоинты отвечают корректно через Nginx
- [ ] Frontend корректно взаимодействует с Backend

#### 3.4. Проверка стабильности

- [ ] Перезапуск одного сервиса не ломает остальные
- [ ] `docker compose restart` — сервисы корректно перезапускаются
- [ ] Нет утечек памяти в логах

### Практика 4: Проверка Dockerfile

DevOps **ОБЯЗАН** для каждого затронутого Dockerfile проверить:

1. **Многоэтапная сборка** (multi-stage build) — для минимизации размера образа
2. **Минимальный базовый образ** — `alpine` или `scratch` где возможно
3. **Нет секретов в образе** — пароли, ключи, токены не попали в образ
4. **Правильный порядок слоёв** — часто меняющиеся слои внизу
5. **.dockerignore** — исключены ненужные файлы
6. **Non-root пользователь** — приложение работает не от root (где возможно)

### Практика 5: Проверка конфигурации

DevOps **ОБЯЗАН** проверить:

1. **docker-compose.yml**:
   - Версия API актуальна
   - Сервисы правильно связаны (depends_on)
   - Порты не конфликтуют
   - Volumes правильно описаны
   - Переменные окружения корректны
   - Restart policy указана
   - Resource limits указаны (где необходимо)

2. **Nginx конфигурация**:
   - Reverse proxy настроен корректно
   - SSL/TLS параметры актуальны
   - Проксирование на правильные порты
   - WebSocket поддержка (если нужна)
   - Gzip сжатие включено

3. **.env файл**:
   - Все переменные из `.env.example` определены
   - Нет реальных секретов в `.env` (только в `.env.example` с placeholder-ами)
   - `.env` добавлен в `.gitignore`

### Практика 6: Логирование и мониторинг

DevOps **ОБЯЗАН**:

1. Проверить что логи сервисов доступны:
   ```bash
   docker compose logs --tail=100 [service]
   ```
2. Убедиться что логи структурированы и читаемы
3. Проверить ротацию логов (log driver настройка)
4. Убедиться что в логах нет секретов (пароли, ключи, токены)

## Анти-паттерны (ЗАПРЕЩЕНО)

1. Не пересобирать сервисы после изменений
2. Игнорировать ошибки в логах контейнеров
3. Не проверять интеграцию между сервисами
4. Пропускать проверку Dockerfile на безопасность
5. Коммитить секреты в docker-compose или Dockerfile
6. Не проверять связку Nginx → Backend → Frontend
7. Принимать нестабильные контейнеры (падающие при старте)
8. Не оформлять итоговый отчёт
9. Оставлять контейнеры с ошибками без внимания
10. Не проверять что .env не содержит реальных секретов
