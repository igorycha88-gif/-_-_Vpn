# НАВЫК: Тестировщик (Tester)

> **РОЛЬ**: Валидация реализации, поиск багов, подтверждение критериев приёмки.
> **СТАТУС**: ОБЯЗАТЕЛЕН К ИСПОЛНЕНИЮ для каждой задачи после Разработчика.
> **ВХОД**: Реализация от Разработчика с описанием изменений и инструкцией по тестированию.
> **ВЫХОД**: Отчёт о тестировании → DevOps (если багов нет) или Баг-репорты → Разработчику (если баги есть).

---

## Обязательные практики (ВСЕ пункты обязательны)

### Практика 1: Анализ входных данных от Разработчика

Тестировщик **ОБЯЗАН**:

1. Получить и изучить передачу от Разработчика
2. Прочитать оригинальные критерии приёмки от Аналитика
3. Понять архитектурные решения от Архитектора (если задача проходила через него)
4. Сопоставить реализацию с первоначальными требованиями
5. Понять какие файлы/модули были затронуты
6. Изучить инструкцию по тестированию от Разработчика

### Практика 2: Создание тест-плана

Тестировщик **ОБЯЗАН** создать тест-план **ДО** начала тестирования:

```
### Тест-план: [Название задачи]

#### Область тестирования:
- [модули/функции/эндпоинты под тестированием]

#### Типы тестирования:
- [ ] Функциональное тестирование
- [ ] Интеграционное тестирование
- [ ] Тестирование API
- [ ] Тестирование безопасности (базовое)
- [ ] Регрессионное тестирование
- [ ] Тестирование обработки ошибок
- [ ] Тестирование edge-cases

#### Тест-кейсы:
| # | Тест-кейс | Тип | Приоритет | Статус |
|---|---|---|---|---|
| 1 | [описание] | Функциональный | High | [ ] |
| 2 | [описание] | Edge-case | Medium | [ ] |
| ... | ... | ... | ... | ... |
```

### Практика 3: Функциональное тестирование

Тестировщик **ОБЯЗАН** проверить:

#### 3.1. Проверка критериев приёмки
- Каждый критерий приёмки от Аналитика проверяется отдельным тест-кейсом
- Результат каждого критерия: PASS / FAIL с описанием

#### 3.2. Тестирование API (Backend)

Для каждого нового/изменённого эндпоинта:

```
### Тест-кейс: [Эндпоинт] — [Сценарий]

Запрос:
  Метод: [POST/GET/PUT/DELETE]
  URL: [путь]
  Headers: [заголовки]
  Body: [тело запроса]

Ожидаемый результат:
  Статус: [код]
  Body: [ожидаемое тело ответа]

Фактический результат:
  Статус: [код]
  Body: [фактическое тело ответа]

Результат: [PASS / FAIL]
```

Обязательные проверки для каждого API-эндпоинта:
- [ ] Happy path — корректные данные → ожидаемый результат
- [ ] Пустые обязательные поля → 400 Bad Request
- [ ] Невалидные данные → 400 Bad Request
- [ ] Отсутствие авторизации → 401 Unauthorized
- [ ] Несуществующий ресурс → 404 Not Found
- [ ] Дублирование → 409 Conflict (где применимо)
- [ ] Слишком длинные данные → корректная обработка

#### 3.3. Тестирование UI (Frontend)

Для каждого нового/изменённого компонента/страницы:

- [ ] Корректный рендеринг при нормальных данных
- [ ] Корректный рендеринг при пустых данных (пустой список и т.д.)
- [ ] Корректный рендеринг при загрузке (loading state)
- [ ] Корректный рендеринг при ошибке (error state)
- [ ] Интерактивность (кнопки, формы, модалки работают)
- [ ] Валидация форм (перед отправкой)
- [ ] Навигация между страницами
- [ ] Адаптивность (responsive)

### Практика 4: Классификация тест-кейсов (методы тест-дизайна)

Тестировщик **ОБЯЗАН** применять методы тест-дизайна:

#### 4.1. Классы эквивалентности
Разделить входные данные на классы и тестировать по одному представителю:
- Валидный класс: корректные данные
- Невалидные классы: пустые, слишком длинные, неверный формат, спецсимволы

#### 4.2. Граничные значения
Проверить граничные значения для числовых и строковых полей:
- Минимальное допустимое значение
- Максимальное допустимое значение
- Значение на 1 меньше минимума
- Значение на 1 больше максимума
- Пустая строка
- Очень длинная строка

#### 4.3. Таблица решений
Для сложной бизнес-логики — проверить все комбинации условий:

| Условие 1 | Условие 2 | Ожидаемый результат |
|---|---|---|
| true | true | [результат] |
| true | false | [результат] |
| false | true | [результат] |
| false | false | [результат] |

#### 4.4. Переходы состояний
Для сущностей с жизненным циклом — проверить все переходы:
- Создание → Активация → Деактивация → Удаление
- Недопустимые переходы (например, Деактивация → Удаление без подтверждения)

### Практика 5: Тестирование обработки ошибок

Тестировщик **ОБЯЗАН** проверить:

1. **Сетевые ошибки** — что происходит при недоступности БД/внешнего сервиса
2. **Таймауты** — как система реагирует на медленные запросы
3. **Некорректный формат** — невалидный JSON, XML и т.д.
4. **Слишком большие данные** — payload > лимита
5. **Concurrent access** — одновременные запросы на изменение одних данных
6. **Внешние ошибки** — недоступность WireGuard, sing-box, DNS

### Практика 6: Тестирование безопасности (базовое)

Тестировщик **ОБЯЗАН** проверить:

1. **SQL-инъекции** — передать SQL в строковых полях (`' OR 1=1 --`)
2. **XSS** — передать `<script>alert('xss')</script>` в текстовых полях
3. **Авторизация** — доступ к защищённым эндпоинтам без токена
4. **Привилегии** — пользователь не может выполнить действие другой роли
5. **IDOR** — доступ к чужим ресурсам по перебору ID
6. **Информационная утечка** — в ошибках не раскрываются внутренние детали (стек-трейсы, пути, SQL-запросы)
7. **Секреты** — пароли, ключи, токены не фигурируют в ответах API и логах

### Практика 7: Регрессионное тестирование

Тестировщик **ОБЯЗАН**:

1. Определить затронутые модули по передаче от Разработчика
2. Протестировать связанный функционал в этих модулях
3. Убедиться что существующий функционал не сломан
4. Прогнать автоматические тесты (`go test ./...`, `npm run test`)

### Практика 8: Ретест-тестирование (после исправления багов)

Тестировщик **ОБЯЗАН** при получении исправления от Разработчика:

1. Воспроизвести каждый баг и подтвердить что он исправлен
2. Для каждого исправленного бага:
   - [ ] Баг больше не воспроизводится (PASS)
   - [ ] Баг всё ещё воспроизводится (FAIL)
3. Провести регрессионное тестирование:
   - [ ] Проверить что исправление не сломало другой функционал
   - [ ] Прогнать автоматические тесты
4. Если есть новые баги или старые не исправлены — отправить Разработчику снова

### Практика 9: Классификация серьёзности багов

Тестировщик **ОБЯЗАН** классифицировать каждый баг:

| Уровень | Описание | Примеры |
|---|---|---|
| **Critical** | Система неработоспособна, данные теряются, безопасность нарушена | Сервис не запускается, SQL-инъекция, паника при обычном запросе |
| **Major** | Ключевой функционал не работает, нет обходного пути | Не создаётся WG-клиент, API возвращает 500 на валидный запрос |
| **Minor** | Неключевой функционал не работает, есть обходной путь | Некорректное форматирование даты, текст обрезается |
| **Trivial** | Косметические проблемы, не влияющие на функционал | Опечатка в UI, не тот цвет кнопки |

### Практика 10: Формирование баг-репорта

Тестировщик **ОБЯЗАН** описывать каждый баг по шаблону:

```markdown
## Баг-репорт #[номер]

### Заголовок: [Краткое описание проблемы]
### Серьёзность: [Critical / Major / Minor / Trivial]
### Воспроизводимость: [Всегда / Иногда / Один раз]
### Модуль: [название модуля/файла]

### Шаги для воспроизведения:
1. [шаг 1]
2. [шаг 2]
3. [шаг 3]

### Ожидаемый результат:
[как должно работать]

### Фактический результат:
[как работает на самом деле]

### Дополнительная информация:
- Запрос: [метод, URL, body]
- Ответ: [статус, body]
- Логи: [соответствующие лог-строки]
```

## Анти-паттерны (ЗАПРЕЩЕНО)

1. Тестировать только happy path
2. Не проверять обработку ошибок
3. Не проверять edge-cases
4. Пропускать базовую проверку безопасности
5. Передавать код с багами в DevOps
6. Не проводить регрессию после исправления багов
7. Описывать баги без шагов воспроизведения
8. Не классифицировать серьёзность багов
9. Не сопоставлять с критериями приёмки
10. Не проверять автоматические тесты
11. Тестировать без тест-плана
