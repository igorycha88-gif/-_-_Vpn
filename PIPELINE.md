# SmartTraffic — Конвейер разработки (Development Pipeline)

> Конвейер определяет обязательный порядок прохождения задач через роли с чёткими правилами передачи, критериями качества и обязательными практиками.

## Роли и их назначение

| Роль | Файл навыков | Назначение |
|---|---|---|
| **Аналитик** | `.opencode/skills/analyst.md` | Анализ требований, декомпозиция, критерии приёмки |
| **Архитектор** | `.opencode/skills/architect.md` | Проектирование архитектуры (для сложных задач) |
| **Разработчик** | `.opencode/skills/developer.md` | Реализация кода, исправление багов |
| **Тестировщик** | `.opencode/skills/tester.md` | Тестирование, валидация, ретест |
| **DevOps** | `.opencode/skills/devops.md` | Сборка, развёртывание, итоги |

## Правила маршрутизации задач

### Правило 1: Вход задачи

Любая задача поступает к **Аналитику**. Аналитик обязан:
1. Проанализировать постановку задачи
2. Определить сложность (простая / средняя / сложная)
3. Выполнить все обязательные практики из своего навыка
4. Передать задачу дальше по конвейеру

### Правило 2: Определение сложности

| Критерий | Простая | Средняя | Сложная |
|---|---|---|---|
| Количество файлов | 1–3 | 4–10 | >10 |
| Новая функциональность | Нет | Частично | Да |
| Затрагивает архитектуру | Нет | Косвенно | Да |
| Требует новый API/эндпоинт | Нет | Возможно | Да |
| Интеграция новых библиотек | Нет | Да | Да |
| Изменение схемы БД | Нет | Возможно | Да |

**Маршрутизация по сложности:**
- **Простая** → Аналитик → Разработчик
- **Средняя** → Аналитик → Архитектор → Разработчик
- **Сложная** → Аналитик → Архитектор → Разработчик

### Правило 3: Обязательный порядок

```
Аналитик → [Архитектор] → Разработчик → Тестировщик → [цикл баг-фикс] → DevOps
```

- Архитектор подключается **ТОЛЬКО** для средних и сложных задач
- Цикл баг-фикс повторяется пока все баги не будут закрыты
- DevOps получает задачу **ТОЛЬКО** после успешного прохождения тестирования

### Правило 4: Критерии передачи между ролями

#### Аналитик → Архитектор / Разработчик
- [ ] Описана постановка задачи
- [ ] Определён контекст (ссылки на файлы, модули)
- [ ] Сформулированы критерии приёмки (Acceptance Criteria)
- [ ] Определены edge-кейсы и граничные условия
- [ ] Оценка рисков и влияния на существующий код
- [ ] Определена сложность задачи

#### Архитектор → Разработчик
- [ ] Описана архитектурная схема изменений
- [ ] Определены интерфейсы и контракты
- [ ] Выбраны паттерны проектирования
- [ ] Описана стратегия обработки ошибок
- [ ] Учтены требования безопасности
- [ ] Определена стратегия тестирования на уровне архитектуры

#### Разработчик → Тестировщик
- [ ] Код реализован согласно спецификации
- [ ] Unit-тесты написаны (покрытие ≥ 80% для нового кода)
- [ ] Линтинг и типизация проходят без ошибок
- [ ] Нет TODO/FIXME/hack в коде
- [ ] Обработка ошибок реализована на всех уровнях
- [ ] Логирование добавлено для ключевых операций
- [ ] Self-review кода выполнен

#### Тестировщик → DevOps (если багов нет)
- [ ] Все тест-кейсы пройдены
- [ ] Критерии приёмки подтверждены
- [ ] Нет критических и major-багов
- [ ] Регрессионное тестирование пройдено
- [ ] Сформирован отчёт о тестировании

#### Тестировщик → Разработчик (если есть баги)
- [ ] Каждый баг описан по шаблону: [Шаги] → [Ожидаемый результат] → [Фактический результат]
- [ ] Указана серьёзность бага (Critical / Major / Minor)
- [ ] Приложены логи/скриншоты при необходимости
- [ ] Указана воспроизводимость (всегда / иногда / один раз)

#### DevOps — финальная стадия
- [ ] Все сервисы пересобраны локально
- [ ] Docker Compose поднимается без ошибок
- [ ] Интеграционная проверка всех сервисов
- [ ] Подведены итоги итерации

## Цикл исправления багов

```
Тестировщик нашёл баги
        │
        ▼
Разработчик получает баг-репорт
        │
        ├── Воспроизводит баг
        ├── Пишет тест, подтверждающий баг (failing test)
        ├── Исправляет код
        ├── Проверяет, что тест проходит
        ├── Прогоняет полный набор тестов (регрессия)
        │
        ▼
Тестировщик получает на ретест
        │
        ├── Подтверждает исправление бага
        ├── Проверяет, что не сломалось другое (регрессия)
        │
        ├── Багов нет ──▶ DevOps
        └── Есть баги  ──▶ снова Разработчику
```

## Формат передачи задачи между ролями

Каждая передача задачи оформляется в следующем формате:

```markdown
## Передача: [Роль-отправитель] → [Роль-получатель]

### Задача: [Название]
### ID задачи: [номер из TZ.md или уникальный ID]

### Контекст:
- [ссылки на файлы, модули, документацию]

### Описание:
[подробное описание для следующей роли]

### Критерии приёмки:
- [ ] [критерий 1]
- [ ] [критерий 2]

### Дополнительная информация:
- [риски, зависимости, замечания]

### Чеклист передачи:
- [ ] [все пункты из правил передачи для данной пары ролей]
```

## Обязательные правила для ВСЕХ ролей

1. **Навыки обязательны к исполнению** — каждая роль обязана следовать всем практикам из своего файла навыка
2. **Контекст проекта** — каждая роль обязана прочитать `TZ.md`, `ARCHITECTURE.md` и `PIPELINE.md` перед началом работы
3. **Чистота кода** — никаких комментариев в коде (кроме случаев явного запроса), никаких TODO/FIXME
4. **Язык** — вся коммуникация на русском языке
5. **Верификация** — перед передачей задачи следующей роли, текущая роль обязана верифицировать свой результат
6. **Обратная связь** — при обнаружении проблем на любом этапе, задача возвращается на предыдущий этап с описанием проблемы
7. **Идемпотентность** — повторный запуск любого этапа должен приводить к тому же результату
